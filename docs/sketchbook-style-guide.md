# Sketchbook Visual Style Guide

This guide defines the visual language for all learning notes in this repository.
It is based on a notebook/whiteboard style: hand-drawn feel, clear structure, and dense visual communication.

## 1) Core style principles

1. **Visual-first, text-second**  
   Explanations should be driven by panels, mini-diagrams, icons, arrows, and labeled boxes.
2. **Human, handwritten tone**  
   Use hand-drawn looking typography and imperfect shapes for warmth.
3. **Free-flow composition with clear anchors**  
   Content can be non-linear, but major sections must still be easy to scan.
4. **Teach through grouping**  
   Break concepts into small "cards" with short bullets and supporting visuals.
5. **Operational clarity**  
   Learner should understand "what it is", "how it works", and "how to use it" at a glance.

## 2) Typography (font recommendations)

Use a combination similar to the provided references:

- **Main title / section headers**: `Permanent Marker`, `Kalam Bold`, or `Balsamiq Sans Bold`
- **Body text**: `Patrick Hand`, `Caveat`, or `Gloria Hallelujah`
- **Technical labels / terminal snippets**: `JetBrains Mono`, `Fira Code`, or `IBM Plex Mono`

### Typography usage rules

- Title should be large and expressive.
- Keep body text short (1-3 lines per block).
- Use color and size differences to create hierarchy instead of long paragraphs.

## 3) Color and line style

- Background: off-white/light paper tone (`#FAF8F2` or similar)
- Primary ink: dark gray/black for text
- Accent colors: blue, green, orange, purple (limited palette, reused consistently)
- Borders: rounded rectangles with hand-drawn style stroke
- Connector lines/arrows: curved where helpful, mostly thin and unobtrusive

## 4) Layout blueprint per note

Each note should include these zones:

1. **Top header**
   - Topic title
   - 1-line learning goal
   - Optional "Day X" or series marker
2. **Main concept panels (2-6 cards)**
   - Each card explains one subsystem/idea
   - Include icon + short bullets + tiny diagram
3. **Flow strip ("How it works")**
   - Left-to-right sequence diagram or pipeline
4. **Quick recap**
   - Key takeaways (3-7 bullets)
   - Optional cheat sheet or glossary
5. **Applied section**
   - Checklist, common mistakes, or debugging lens

## 5) Content density rules

- Prefer **many small blocks** over one large block.
- Keep bullets concise and action-oriented.
- Avoid walls of text.
- Maximum 30% prose, minimum 70% visual/diagrammatic structure.

## 6) Visual components library (logical)

Use these recurring visual components in every topic:

- Section card
- Sticky-note callout
- Arrow connector
- Mini architecture diagram
- Timeline or sequence strip
- Checklist row
- "Key takeaway" badge
- Terminal/API snippet block

## 7) Voice and pedagogy

- Tone: practical, clear, friendly, not fluffy.
- Explain concepts as systems and interactions.
- Always include operational reasoning:
  - Why this part exists
  - What breaks if it fails
  - What to monitor

## 8) Definition of done for each note

A note is complete when it has:

- A visual-first layout with clearly separated panels
- At least one end-to-end flow/sequence visual
- A recap section with key takeaways
- A practical checklist or failure-mode section
- Consistent style (fonts, palette, icon language)
